What are and not Microservices.

Are:

"Microservices are small, autonomous services that work together, modelled around a business domain."
Sam Newman


Not:

Monolith application is always built as a single, autonomous unit.

  • Explain why the word Architect is wrong (not right) when comes to building Microservices
  • Explanation of Monolith like a single building that handles a lot of operations. All of them in same place.
  • Explanation of Micorservices like town and how each building handles only one operation and is easy to maintain it

We create microservices around business capabilities. Single business logic. Scaling. Deploying each services independently

A modification made to a small section of an application might require building and deploying an entirely new version. Deploying all together.

Challenges with monolith.

  • Difficult to scale.
  • Architecture is hard to maintain.
  • New releases takes long.


Delivery pipeline

  • Build > Test > Release

That is because in monolith there is too much coupling. Share libraries, Share DB, one depends to on the other

Benefits of Microservices

  • Easy to scale individual microservices
  • Clear ownership
  • New release are really fast takes only minutes


Delivery pipeline is really quick.

  • Build > Test > Release

Principles of Microservices.

  1. Create Microservices around business capabilities.
  2. Microservices only rely on each other's public API
  3. Hide your data.
  4. Deploy independently.
  5. Decentralization.
  6. Keep each Microservice on its own host.
  1. We've talked already about it. That we create a microservice around single business logic
  2. That means they communicate to each other only via those APIs.
  3. Microservice A should not know about microservice B insides.
  4. To be able to deploy a micoservice independently without of needing to change anything else outside of it.
  5. Monolithic implementations usually have a single logical database.... With microservices, each service can manage its own database
